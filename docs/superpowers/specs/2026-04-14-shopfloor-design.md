# Shopfloor v0.1 Design

**Status:** Draft, pending review
**Date:** 2026-04-14
**Author:** Brainstormed collaboratively with Claude (Opus 4.6)
**Supersedes:** N/A (initial design)

## 1. Overview

Shopfloor is a reusable GitHub workflow that wraps `anthropics/claude-code-action@v1` to drive a staged, human-gated software delivery pipeline. A single issue progresses through optional **triage**, optional **spec**, optional **plan**, mandatory **implementation**, and mandatory **agent review** stages. Each document stage produces its own pull request, humans review and merge each PR, and Shopfloor advances the state machine to the next stage on merge. Smaller tickets skip stages. Every implementation PR additionally passes through an automated multi-reviewer loop before a human is asked to merge.

GitHub itself is the state machine. Labels carry state, branch names carry linkage, PR bodies carry machine-readable metadata, and markdown files in `docs/shopfloor/` carry the artifacts (specs and plans) as durable, reviewable, version-controlled documents. Shopfloor holds no state of its own. There is no backend service, no database, no web UI.

### One-sentence pitch

Shopfloor turns a GitHub issue into a reviewed spec, a reviewed plan, and a reviewed implementation (each on its own PR, each approved by a human) by orchestrating `claude-code-action` across the stages. Users add one workflow file. That is the entire install.

## 2. Goals and non-goals

### Goals

- **Opinionated multi-stage pipeline** that codifies a spec → plan → implement workflow on top of `claude-code-action` without forking it.
- **GitHub-native UX**: every artifact, state transition, and review happens in the GitHub UI the user already knows. No third-party dashboard, no Slack-only interactions, no CLI that duplicates GitHub's review features.
- **Human-in-the-loop at every stage boundary**: no stage begins until the previous stage's PR is merged (or skipped because the ticket is small enough).
- **Automated multi-reviewer pass before human review**: every implementation PR is reviewed by a matrix of specialised agent reviewers (CLAUDE.md compliance, bugs, security, code smells). The review loops with the implementation agent until the reviewer is clean or a configurable iteration cap is reached, then surfaces a commit status for human merge.
- **Durable artifacts**: specs and plans are committed markdown files, reviewable as normal PR diffs, permanent unless the repo owner deletes them.
- **Single-file install** for consumers: one `.github/workflows/shopfloor.yml` caller file, no scaffolding CLI, no config file sprawl.
- **Zero trust of the LLM in the state machine**: every label change, PR creation, issue close, and branch advance is performed by deterministic code. The agent only produces content.
- **Uniform comment voice**: all user-visible comments are templated by Shopfloor code, not generated by the agent.
- **Full pass-through of `claude-code-action` auth modes**: Anthropic API key, Claude Code OAuth token, Bedrock, Vertex, Foundry, custom GitHub App.

### Non-goals (v0.1)

These appear in the originating vision but are deliberately out of scope for the first release:

- Sandboxed execution beyond what GitHub Actions runners already provide.
- OpenTelemetry export, observability dashboards, or cost tracking.
- Slack, email, or other external notification integrations.
- Multi-agent parallelism beyond the review stage's fixed 4-reviewer matrix (e.g., arbitrary user-defined reviewer ensembles, dynamically sized review pools).
- A "curated capabilities" or shared skill library beyond what `claude-code-action` ships with.
- Automatic complexity re-classification mid-pipeline.
- Budget gates or per-run spend limits.
- A `shopfloor init` CLI or scaffolding PR.
- A hosted marketplace GitHub App.
- Downstream agents (spec, plan, impl) requesting clarification from humans. Only triage is permitted to ask questions.

Each of these may become a follow-up project after v0.1 ships and survives contact with real use.

## 3. High-level architecture

Shopfloor is, physically:

1. **A reusable GitHub workflow** (`.github/workflows/shopfloor.yml` inside this repository) consumed by users via `uses:` in their own caller workflow.
2. **A small JavaScript/TypeScript router action** (`router/`) that determines which stage to run for a given event and performs all deterministic state mutations (label flips, PR creation, issue close, diagnostic comments).
3. **A small TypeScript MCP server** (`mcp-servers/shopfloor-mcp/`) that exposes Shopfloor-owned tools to the implementation agent, starting with `update_progress` and extensible to more tools in future versions.
4. **Four stage prompt templates** (`prompts/triage.md`, `prompts/spec.md`, `prompts/plan.md`, `prompts/implement.md`), each shipping the system prompt, context placeholder scheme, and structured output instructions for its stage.
5. **A handful of small helper actions or composite steps** under `router/` that the reusable workflow calls to perform specific operations (open stage PR, advance state, report failure, create initial progress comment).

Every component is versioned in the Shopfloor repository. Users reference a specific `@v1` (or later) tag in their caller workflow, and upgrades are a version bump.

### What Shopfloor does not have

- No hosted service.
- No database or persistent storage of its own. Everything it needs is either in the GitHub event payload, on the issue/PR via the GitHub API, or committed as a file in the user's repository.
- No cron or scheduler. Shopfloor only runs in response to GitHub events on the user's repository.

## 4. The state machine

### States

Every Shopfloor-managed issue is in exactly one state at a time, represented by exactly one `shopfloor:*` state label. The states are:

| Label | Meaning |
|---|---|
| `shopfloor:triaging` | Triage agent is reading the issue and deciding what to do. |
| `shopfloor:awaiting-info` | Triage posted clarifying questions; pipeline is paused until a human replies and a maintainer removes this label. |
| `shopfloor:quick` | Triage classified this as a quick fix. Will skip to implementation. |
| `shopfloor:medium` | Triage classified this as medium. Will skip spec, go to plan. |
| `shopfloor:large` | Triage classified this as large. Full spec → plan → impl flow. |
| `shopfloor:needs-spec` | Ready for the spec agent to run. |
| `shopfloor:spec-in-review` | Spec PR is open, awaiting human review and merge. |
| `shopfloor:needs-plan` | Spec has been merged (or skipped). Ready for plan agent. |
| `shopfloor:plan-in-review` | Plan PR is open, awaiting human review and merge. |
| `shopfloor:needs-impl` | Plan has been merged (or skipped). Ready for implementation agent. |
| `shopfloor:needs-review` | Implementation agent has pushed commits to the impl PR; review stage is queued or running. |
| `shopfloor:review-requested-changes` | Review stage produced actionable findings at or above the confidence threshold; implementation agent will re-run in revision mode on next event cycle. |
| `shopfloor:review-approved` | Review stage passed. Commit status `shopfloor/review` is `success`. Awaiting human review and merge. |
| `shopfloor:review-stuck` | The impl↔review loop has hit `max_review_iterations` without converging. Requires a human to intervene by removing the label or taking over the PR. |
| `shopfloor:impl-in-review` | Equivalent of `shopfloor:review-approved` for tickets that explicitly bypass review via `shopfloor:skip-review`. Implementation PR awaits human merge without an agent review pass. |
| `shopfloor:done` | Implementation PR has been merged. Terminal state. Original issue auto-closed. |

Plus these orthogonal labels that can appear on top of any state label:

| Label | Meaning |
|---|---|
| `shopfloor:failed:<stage>` | The last run of `<stage>` failed. A maintainer removes this label to retry. No auto-retry. `<stage>` is one of `triage`, `spec`, `plan`, `implement`, or `review`. |
| `shopfloor:revise` | Manual escape hatch: retrigger the current stage. Used when `Request changes` was not submitted but a human wants the agent to run again. Consumed on trigger (label removed after firing). |
| `shopfloor:skip-review` | Applied to an impl PR (or to the issue) to completely bypass the agent review stage for this ticket. Useful when a human wants to take over the PR manually. |

### Transitions

The core flow, with all edges:

```
(new issue, no shopfloor label)
   │ issue opened → router assigns shopfloor:triaging
   ▼
shopfloor:triaging
   │ triage agent runs
   ├──[status: needs_clarification]──► shopfloor:awaiting-info
   │                                          │ maintainer removes label
   │                                          └──► shopfloor:triaging
   │
   └──[status: classified, complexity: quick]────► shopfloor:quick + shopfloor:needs-impl
   └──[status: classified, complexity: medium]───► shopfloor:medium + shopfloor:needs-plan
   └──[status: classified, complexity: large]────► shopfloor:large + shopfloor:needs-spec

shopfloor:needs-spec
   │ spec agent runs, writes docs/shopfloor/specs/<n>-<slug>.md, pushes, router opens PR
   ▼
shopfloor:spec-in-review
   │ Request changes / shopfloor:revise → spec agent re-runs on same branch
   │ PR approved and merged → router transitions
   ▼
shopfloor:needs-plan
   │ plan agent runs, writes docs/shopfloor/plans/<n>-<slug>.md
   ▼
shopfloor:plan-in-review
   │ Request changes / shopfloor:revise → plan agent re-runs
   │ PR approved and merged → router transitions
   ▼
shopfloor:needs-impl
   │ router creates initial progress comment, impl agent runs,
   │ impl agent pushes commits to the impl PR branch
   ▼
(pull_request.synchronize on the impl PR)
   ▼
shopfloor:needs-review
   │ review stage runs (matrix: compliance | bugs | security | smells)
   │ aggregator job collects results, dedupes, filters by confidence
   │
   ├──[filtered comments empty, all verdicts "clean"]──► shopfloor:review-approved
   │                                                       - commit status shopfloor/review = success
   │                                                       - approving GitHub review posted
   │                                                       - PR comment: "Agent review clean. Ready for human review."
   │                                                       - iteration counter frozen
   │                                                       │
   │                                                       ▼
   │                                                    human reviews and merges
   │                                                       ▼
   │                                                    shopfloor:done
   │
   └──[filtered comments non-empty]───────────────────► shopfloor:review-requested-changes
                                                          - commit status shopfloor/review = failure
                                                          - Request Changes review with batched inline comments
                                                          - iteration counter incremented in PR body metadata
                                                          │
                                                          │ Request Changes review fires
                                                          │ pull_request_review.submitted with
                                                          │ changes_requested → router dispatches
                                                          │ impl stage in revision mode
                                                          ▼
                                                       shopfloor:needs-impl (revision mode)
                                                          │
                                                          │ if counter > max_review_iterations:
                                                          │   router aborts the loop instead
                                                          │   → shopfloor:review-stuck
                                                          │
                                                          └──► loops back to impl execution

shopfloor:review-stuck
   │ human intervention required
   │ human removes the label to retry, or
   │ human takes over the PR manually, or
   │ human merges despite the unresolved findings
   │ (no automatic exit)
```

The `shopfloor:skip-review` label short-circuits the review loop: if the impl PR (or its origin issue) carries this label, the router transitions directly from `shopfloor:needs-impl` + impl agent completion to `shopfloor:impl-in-review` without running the review stage at all. This is the only legitimate way for a Shopfloor impl PR to reach a mergeable state without an agent review.

### Universal abort

If the origin issue is closed by a human at any point, the router refuses to advance the pipeline on subsequent events. In-flight stage PRs are left untouched for humans to close or keep. Closing the issue is the universal kill signal for Shopfloor and requires no other action.

### Concurrency

- A GitHub Actions `concurrency` key scoped per issue (`shopfloor-issue-${{ issue_number }}`) with `cancel-in-progress: false` ensures at most one stage job runs at a time for a given issue. Rapid-fire events on the same issue serialize naturally.
- Different issues run in parallel with no coordination.

## 5. Execution stages

All four stages share a common shape: the router resolves the stage from the event, the stage job runs the agent with a stage-specific prompt and toolset, the agent produces structured output, and the router performs the deterministic post-work (opens PR, flips label, etc.). Each stage differs only in prompt, tool allowlist, and post-work.

### 5.1 Triage stage

**When it runs:** When an issue is opened without a `shopfloor:*` label, or when the `shopfloor:awaiting-info` label is removed after a human reply.

**Agent capabilities:**

- `--allowedTools "Read,Glob,Grep,WebFetch"`
- No shell, no edits, no writes. Triage only reads.

**Purpose:** Decide whether the issue has enough information to proceed, and if so, classify it as `quick`, `medium`, or `large`.

**Prompt inputs:**

- The origin issue body
- All issue comments to date
- Information about the repository (README, top-level structure, any CLAUDE.md or AGENTS.md files)

**Structured output schema:**

```json
{
  "type": "object",
  "properties": {
    "status": { "enum": ["classified", "needs_clarification"] },
    "complexity": { "enum": ["quick", "medium", "large"] },
    "rationale": { "type": "string" },
    "clarifying_questions": {
      "type": "array",
      "items": { "type": "string" }
    }
  },
  "required": ["status"]
}
```

**Router post-work:**

- If `status == "needs_clarification"`: post a single templated comment on the issue containing the clarifying questions, apply `shopfloor:awaiting-info`, remove `shopfloor:triaging`.
- If `status == "classified"`: post a short comment on the issue announcing the classification and rationale, apply the complexity label (`shopfloor:quick|medium|large`), remove `shopfloor:triaging`, and apply the appropriate next-stage label (`shopfloor:needs-spec`, `shopfloor:needs-plan`, or `shopfloor:needs-impl`) based on complexity.

### 5.2 Spec stage

**When it runs:** When an issue carries `shopfloor:needs-spec` (set by triage or by a human override).

**Agent capabilities:**

- `--allowedTools "Read,Glob,Grep,Edit,Write"`
- No shell. Spec writing is thinking and drafting, not execution.

**Purpose:** Produce a reviewable design document for the feature or fix described in the issue.

**Prompt inputs:**

- The origin issue body and comments
- Rationale from triage (if available)
- Repository context (CLAUDE.md/AGENTS.md, README, top-level structure, existing specs in `docs/shopfloor/specs/` for style)
- Target file path: `docs/shopfloor/specs/<issue-number>-<slug>.md`
- Target branch name: `shopfloor/spec/<issue-number>-<slug>`

**Expected agent actions:**

1. Create the target branch from the base branch.
2. Write the spec file to the target path.
3. Commit and push.
4. Return structured output.

**Structured output schema:**

```json
{
  "type": "object",
  "properties": {
    "file_path": { "type": "string" },
    "pr_title": { "type": "string" },
    "pr_body": { "type": "string" },
    "summary_for_issue_comment": { "type": "string" }
  },
  "required": ["file_path", "pr_title", "pr_body"]
}
```

**Router post-work:**

- Open a pull request from the branch to the base branch with the agent-supplied title and body, appending a metadata block:
  ```
  ---
  Shopfloor-Issue: #<n>
  Shopfloor-Stage: spec
  ```
- Apply `shopfloor:spec-in-review` to the issue.
- Post a short comment on the origin issue: "Spec drafted: #<pr-number>. Please review."

### 5.3 Plan stage

**When it runs:** When an issue carries `shopfloor:needs-plan`.

**Agent capabilities:**

- `--allowedTools "Read,Glob,Grep,Edit,Write,Bash(git log:*),Bash(git diff:*),Bash(git show:*)"`
- Read-only git to inspect history and understand context. No mutating shell.

**Purpose:** Produce a step-by-step implementation plan, grounded in the merged spec (if one exists) and the current state of the codebase.

**Prompt inputs:**

- The origin issue body and comments
- The merged spec file contents, if one exists at `docs/shopfloor/specs/<n>-<slug>.md`
- Repository context
- Target file path: `docs/shopfloor/plans/<issue-number>-<slug>.md`
- Target branch name: `shopfloor/plan/<issue-number>-<slug>`

**Structured output schema:** Same as spec stage (`file_path`, `pr_title`, `pr_body`, `summary_for_issue_comment`).

**Router post-work:** Same as spec stage (open PR, metadata block with `Shopfloor-Stage: plan`, apply `shopfloor:plan-in-review`).

### 5.4 Implementation stage

**When it runs:** When an issue carries `shopfloor:needs-impl`.

**Agent capabilities:**

- `--allowedTools "Read,Glob,Grep,Edit,Write,Bash(<user allowlist>),Bash(git log:*),Bash(git diff:*),mcp__shopfloor__update_progress"`
- Full editing and writing of source files.
- Shell access restricted to a user-supplied allowlist (default: `pnpm install`, `pnpm test:*`, `pnpm lint:*`, `pnpm build`, `pnpm exec tsc`, plus read-only git).
- Progress tool for updating the tracking comment.

**Purpose:** Make the actual code changes described in the plan (or in the issue directly, for `quick` tickets).

**Prompt inputs:**

- The origin issue body and comments
- The merged plan file contents, if one exists at `docs/shopfloor/plans/<n>-<slug>.md`
- The merged spec file contents, if one exists
- Repository context
- A pointer to the progress tracking comment ID (also injected as `SHOPFLOOR_COMMENT_ID` env var into the MCP server)
- Target branch name: `shopfloor/impl/<issue-number>-<slug>`

**Router pre-work** (before the agent runs):

1. Create a new branch from the base branch.
2. Post an initial "Implementation starting. I will update this comment with progress." comment on the target branch (as a PR is not yet open, this goes on the origin issue until the PR exists, then the router opens the PR early with a placeholder body and repoints the comment to the PR).

   **Design choice:** To keep all progress visible on the PR being built (not on the issue), the router opens the implementation PR *before* the agent begins work, with a placeholder body ("Shopfloor is drafting this PR now..."). The initial progress comment is posted on that PR, and its ID is passed to the agent. Once the agent finishes, the router updates the PR body with the agent's final structured output.

3. Write an MCP config file to `$RUNNER_TEMP/shopfloor-mcp.json` declaring the `shopfloor` server with the comment ID injected:

   ```json
   {
     "mcpServers": {
       "shopfloor": {
         "command": "bun",
         "args": ["run", "${{ github.action_path }}/mcp-servers/shopfloor-mcp/index.ts"],
         "env": {
           "GITHUB_TOKEN": "***",
           "REPO_OWNER": "<owner>",
           "REPO_NAME": "<repo>",
           "SHOPFLOOR_COMMENT_ID": "<id>",
           "GITHUB_API_URL": "<url>"
         }
       }
     }
   }
   ```

4. Pass `--mcp-config $RUNNER_TEMP/shopfloor-mcp.json` in the impl stage's `claude_args`.

**Expected agent actions:**

1. On startup, call `update_progress` with an initial TODO-style checklist derived from the plan.
2. Work through the plan, updating the progress comment as tasks complete.
3. Commit and push incrementally (or as one final commit; agent choice).
4. Return structured output.

**Structured output schema:**

```json
{
  "type": "object",
  "properties": {
    "pr_title": { "type": "string" },
    "pr_body": { "type": "string" },
    "summary_for_issue_comment": { "type": "string" },
    "changed_files": { "type": "array", "items": { "type": "string" } }
  },
  "required": ["pr_title", "pr_body"]
}
```

**Router post-work:**

- Update the PR with the agent-supplied title and body, appending the metadata block (`Shopfloor-Issue: #<n>`, `Shopfloor-Stage: implement`).
- Update the progress tracking comment one final time with a "done" marker, whether the agent updated it last or not. This ensures a consistent terminal state even if the agent crashed mid-run.
- Check for `shopfloor:skip-review` on the impl PR or the origin issue. If present, apply `shopfloor:impl-in-review` and skip the review handoff. If absent, apply `shopfloor:needs-review`. See section 5.5 for the full trigger semantics and the skip-review branch contract.
- Post "Implementation drafted: #<pr-number>" on the origin issue.

### 5.5 Review stage

**Trigger (authoritative):** The review stage runs when the router's stage resolution outputs `stage=review`. The router outputs `stage=review` under exactly these conditions:

1. The event is `pull_request.synchronize` on a PR whose body contains `Shopfloor-Stage: implement`, AND
2. The origin issue (parsed from the `Shopfloor-Issue: #N` metadata in the PR body) is not closed, AND
3. Neither the PR nor the origin issue carries the `shopfloor:skip-review` label, AND
4. The skip-condition gate (section 5.5.1) passes, AND
5. The current iteration counter in the PR body metadata is less than or equal to `max_review_iterations`. (The `<=` is deliberate. The run-entry gate admits the stage; the post-run guard in section 5.5.3 step 10 is what actually caps the loop, so the final allowed iteration must pass this gate to reach the guard.)

OR:

6. The event is `issues.unlabeled` and the removed label is `shopfloor:review-stuck`, in which case the router dispatches review against the current HEAD of the associated impl PR (a maintainer manually unsticking a capped loop).

Applying or removing `shopfloor:needs-review` by hand is **not** a review trigger. The label exists as a visual state indicator only; the router applies it as part of the review stage's entry and clears it when the verdict is reached. Human label changes on `needs-review` are ignored by the router.

The impl stage's post-work (section 5.4's router post-work) is responsible for applying `shopfloor:needs-review` after impl completes, **unless** the `shopfloor:skip-review` label is present on the PR or issue at the moment impl post-work runs. In that case, impl post-work skips `needs-review` entirely and applies `shopfloor:impl-in-review` directly, and the subsequent `pull_request.synchronize` event still fires but the router's condition (3) filters it out, so no review stage runs.

**Purpose:** Perform an automated, multi-perspective code review of the implementation changes. Either approve the PR for human review or request changes that re-trigger the implementation agent.

**Shape:** The review stage is a **4-way matrix job** plus a **single aggregator job**. The matrix fans out four independent `claude-code-action` invocations, one per reviewer perspective, running in parallel on separate runners. Each produces structured output via `--json-schema`. The aggregator job runs only after all four matrix cells complete (`needs: [review-compliance, review-bugs, review-security, review-smells]`), collects their outputs, performs deduplication and filtering, and posts the final GitHub review and commit status.

#### 5.5.1 Skip conditions

The review stage performs a pre-check at the start of the stage job. The review is skipped when any of the following is true:

1. The PR is closed or already merged.
2. The PR is in draft state.
3. The PR has zero changed files.
4. The PR's changed files are entirely within `docs/shopfloor/` (this is a spec or plan PR, not an implementation PR).
5. The PR already has a Shopfloor-authored GitHub review at the current HEAD SHA (router checks via `GET /repos/:owner/:repo/pulls/:pr/reviews`, looks for reviews whose body begins with a Shopfloor marker and whose `commit_id` matches HEAD).
6. The PR carries the label `shopfloor:skip-review`.
7. The issue-level origin carries `shopfloor:skip-review`.

When skipped, the router posts a one-line comment on the PR (only if not already posted at this SHA) explaining the skip reason, and emits `stage=none` so no downstream jobs run.

#### 5.5.2 The four reviewer subagents

Each subagent runs as a matrix cell in the review stage job. All four share the same repository checkout, the same base tool allowlist, and the same schema for structured output. They differ only in the prompt file they use, the model they run on, and (optionally) their max turns.

| Reviewer | Prompt | Default model | Reviews for |
|---|---|---|---|
| `compliance` | `prompts/review-compliance.md` | `claude-opus-4-6` | Violations of `CLAUDE.md`, `AGENTS.md`, `GEMINI.md`, `CONTRIBUTING.md`, and any similar in-repo guideline files that apply to the changed paths. |
| `bugs` | `prompts/review-bugs.md` | `claude-opus-4-6` | Obvious bugs, missed requirements from the linked spec/plan, off-by-one errors, null deref risks, incorrect error handling. |
| `security` | `prompts/review-security.md` | `claude-opus-4-6` | Injection vulnerabilities, auth/authz bypass, secret leakage, unsafe deserialisation, path traversal, insecure defaults. |
| `smells` | `prompts/review-smells.md` | `claude-opus-4-6` | Code smells, dead code, duplication, overly broad abstractions, unclear names, premature generalisation, excessive comments. |

Each reviewer is assigned its own tunable inputs on the reusable workflow:

- `review_<name>_model` (default: `claude-opus-4-6`)
- `review_<name>_max_turns` (default: `8`)
- `review_<name>_enabled` (default: `true`; setting false drops that matrix cell entirely)

This lets users run cheaper models for low-signal reviewers (e.g., smells on Haiku) and keep high-signal reviewers on Opus.

**Agent capabilities (shared across all four reviewers):**

- `--allowedTools "Read,Glob,Grep,Bash(git diff:*),Bash(git log:*),Bash(git show:*),WebFetch"`
- Read-only. No `Edit`, no `Write`, no mutating bash, no comment or PR MCP tools. The reviewers cannot modify anything.

**Prompt inputs (shared):**

- The diff of the impl PR (from `git diff $BASE...HEAD`, trimmed to sensible size).
- The list of changed file paths.
- The merged spec contents, if any, from `docs/shopfloor/specs/<n>-*.md`.
- The merged plan contents, if any, from `docs/shopfloor/plans/<n>-*.md`.
- The origin issue body and comments (for context).
- The iteration counter (so the reviewer knows whether this is a first-pass or a follow-up).
- Any previously posted Shopfloor review comments on prior iterations, so the reviewer can check whether earlier findings were addressed.
- Reviewer-specific context: compliance reviewer gets the content of CLAUDE.md files; others get specific guidance per prompt.

**Structured output schema (shared):**

```json
{
  "type": "object",
  "properties": {
    "verdict": { "enum": ["clean", "issues_found"] },
    "summary": { "type": "string" },
    "comments": {
      "type": "array",
      "items": {
        "type": "object",
        "properties": {
          "path": { "type": "string" },
          "line": { "type": "integer" },
          "side": { "enum": ["LEFT", "RIGHT"] },
          "start_line": { "type": "integer" },
          "start_side": { "enum": ["LEFT", "RIGHT"] },
          "body": { "type": "string" },
          "confidence": { "type": "integer", "minimum": 0, "maximum": 100 },
          "category": { "enum": ["compliance", "bug", "security", "smell"] }
        },
        "required": ["path", "line", "side", "body", "confidence", "category"]
      }
    }
  },
  "required": ["verdict", "summary", "comments"]
}
```

Field semantics:

- `verdict`: `clean` when the reviewer has nothing actionable to say; `issues_found` otherwise. If `verdict == "clean"`, `comments` should be empty (aggregator enforces this).
- `line` + `side`: single-line comment. `side: "RIGHT"` for the post-change line (the usual case when commenting on added code); `side: "LEFT"` for the pre-change line (commenting on deleted code).
- `start_line` + `start_side`: optional, for multi-line range comments. When set, `line` is the end of the range and `start_line` is the beginning.
- `confidence`: the reviewer's self-reported confidence that this comment is actionable and correct. Used for filtering.
- `category`: must match the reviewer's perspective (`compliance` from the compliance reviewer, etc.). The aggregator verifies this to catch reviewers going out of scope.

#### 5.5.3 The aggregator job

**Trigger:** `needs: [review-compliance, review-bugs, review-security, review-smells]` with `if: always()`, so it runs even if some matrix cells failed (we want to aggregate whatever we got).

**Steps:**

1. Collect each matrix cell's `structured_output` from job outputs. A failed cell contributes nothing (counts as "no findings from this reviewer"). The aggregator logs the cell's failure so it is visible in the run logs.
2. Validate each output against the reviewer schema. Invalid outputs are logged and treated as empty.
3. Concatenate all `comments` arrays. Attach the originating reviewer's name as a new field `source`.
4. **Dedupe** by a stable hash of `{path, line, side, body-content-hash}`. When two reviewers flag the same file+line with substantively similar text, keep the one with the higher confidence. Substantive similarity is computed with a simple token-overlap heuristic (>= 75% overlap on the first 200 characters of the body).
5. **Filter** by `confidence >= review_confidence_threshold` (default: 80).
6. Determine the verdict:
   - **Clean:** the filtered list is empty AND every reviewer that completed successfully returned `verdict == "clean"`.
   - **Issues found:** otherwise.
7. Set the commit status `shopfloor/review` to `pending` at the start of step 1, then update it to `success` or `failure` based on the verdict.
8. Post a GitHub review:
   - **Clean:** `POST /repos/:owner/:repo/pulls/:pr/reviews` with `event: "APPROVE"`, body containing the Shopfloor marker and a one-paragraph summary from the reviewers' `summary` fields, no comments array.
   - **Issues found:** `POST /repos/:owner/:repo/pulls/:pr/reviews` with `event: "REQUEST_CHANGES"`, body containing the Shopfloor marker and a brief summary of which categories produced findings, `comments` array containing every filtered comment (each with `path`, `body`, and `line`+`side` or `start_line`+`start_side`+`line`+`side` for multi-line).
9. **Increment the iteration counter only on `issues_found` verdicts.** The counter tracks impl↔review bounces, which only happen when the review requests changes. On a `clean` verdict the counter is left as-is (the loop terminates naturally). Format: a line `Shopfloor-Review-Iteration: <N>` that the router maintains at the bottom of the PR body. A missing counter is treated as `0`.
10. If the verdict is `issues_found` AND the *incremented* counter would exceed `max_review_iterations` (default: 3), the aggregator does NOT post the Request Changes review (which would trigger another impl run). Instead, it:
    - Applies `shopfloor:review-stuck` to the origin issue.
    - Posts a comment on the PR explaining that the iteration cap was reached, linking to the findings, and asking a human to take over.
    - Sets the commit status to `failure` with a `shopfloor/review: iteration cap reached` description.
    - Emits no Request Changes review.
11. Apply the appropriate label to the origin issue: `shopfloor:review-approved` on clean, `shopfloor:review-requested-changes` on issues found (unless capped), `shopfloor:review-stuck` on cap reached.

**Aggregator tool access:** This is a router-owned step implemented as a composite/JS helper action (`router/aggregate-review/`). It has no LLM. It reads matrix outputs, computes the verdict, and calls the GitHub API directly.

#### 5.5.4 Impl↔review loop

When the aggregator posts a `REQUEST_CHANGES` review, the event `pull_request_review.submitted` fires on the impl PR. The router reads this event, sees that it's a Shopfloor-authored review with `changes_requested` state, and dispatches the impl stage in revision mode. The impl agent re-runs with additional prompt inputs:

- The diff it produced last time
- The Shopfloor review comments (parsed from the review body and `comments` array)
- The previous iteration count (so it knows this is a response to agent review, not to human review)

The impl agent pushes revision commits on the existing branch. The `pull_request.synchronize` event fires, triggering the review stage again. Loop continues until:

- The reviewer is clean, OR
- The iteration cap is reached.

#### 5.5.5 Interaction with human reviews

A human may submit their own review (approving or requesting changes) on an impl PR at any point, including while the agent review loop is still running. Conflict-resolution rules:

- **Human approval** supersedes agent review for merge purposes (the branch protection rule is the ultimate gate). Humans may merge even if `shopfloor/review` is `failure`.
- **Human Request Changes** is also a revision trigger (same as agent Request Changes), but the router distinguishes between human and Shopfloor reviews by author identity. Human revision triggers do NOT increment the iteration counter (the counter only tracks agent↔impl bounces).
- If a human comments while the review stage is running, the concurrency key serializes the next event after the current stage completes.

### 5.6 Stage revision (Request changes / shopfloor:revise)

All three document-producing stages (spec, plan, impl) support revision. The trigger is either:

- A GitHub review on the stage PR with status `changes_requested`, or
- The `shopfloor:revise` label applied to the stage PR.

On revision trigger, the router dispatches the same stage again, but with additional prompt inputs:

- The current (unrevised) spec/plan/impl file contents.
- All PR review comments (line comments and review bodies) since the last run.

The agent works on the existing branch, pushing new commits (never force-pushing) so that existing review threads stay anchored to original lines. The router does not open a new PR; it comments on the existing one: "Revisions pushed. Please re-review." The label stays at `shopfloor:spec-in-review` (or `plan-in-review` or `impl-in-review`).

The `shopfloor:revise` label is removed by the router immediately on trigger so that re-adding it can retrigger cleanly.

### 6.2 The router action

Path: `router/` in the Shopfloor repository.

A small TypeScript action using `@actions/github`. Pure logic: it reads the GitHub event payload, the current labels, linked PRs, branch names, and PR body metadata, and produces exactly one primary output: `stage` (one of `triage`, `spec`, `plan`, `implement`, `review`, `none`), plus ancillary outputs (`issue_number`, `complexity`, `branch_name`, `spec_file_path`, `plan_file_path`, `revision_mode`, `review_comments_json`, `review_iteration`, `impl_pr_number`).

**No LLM calls in the router.** This is the state machine, and state machines must be deterministic to debug.

The router is split into:

- **`src/state.ts`**: Pure functions that take `{event, labels, linkedPrs}` and return `{stage, context}`. Fully unit-testable with no GitHub API dependency. This is where the state machine logic lives.
- **`src/github.ts`**: Thin adapter around `@actions/github`'s `octokit`. Reads labels, reads PR metadata, reads comments. All I/O goes through this module so the state logic stays pure.
- **`src/index.ts`**: Entry point. Parses event, calls `github.ts` to gather state, calls `state.ts` to decide, writes outputs.
- **`src/types.ts`**: Shared types.

The router also ships several subsequent helper actions used by the stage jobs after the agent completes:

- **`router/open-stage-pr/`**: Opens a PR with correct metadata block, given branch name, title, body, and stage.
- **`router/advance-state/`**: Flips labels from `from_label` to `to_label` on an issue or PR.
- **`router/report-failure/`**: Runs in `if: failure()`, posts a diagnostic comment on the relevant artifact, applies `shopfloor:failed:<stage>`.
- **`router/create-progress-comment/`**: Creates the initial tracking comment for the impl stage, returns its ID.
- **`router/finalize-progress-comment/`**: Updates the tracking comment to its final "done" state.
- **`router/handle-merge/`**: Called in the route job when a stage PR merge event is detected. Parses metadata, flips labels, dispatches the next stage.
- **`router/bootstrap-labels/`**: First-run idempotent label creation.
- **`router/aggregate-review/`**: Runs after the review stage's matrix jobs complete. Collects each matrix cell's structured output, dedupes comments, filters by confidence, posts the final GitHub review (APPROVE or REQUEST_CHANGES), updates the commit status `shopfloor/review`, manages the iteration counter, and applies review-outcome labels. No LLM calls.
- **`router/check-review-skip/`**: Evaluates the review stage's skip conditions at the top of the review job. Returns a decision and a reason; if skipping, the stage exits cleanly and posts an explanatory comment.

All helpers share the logic in `router/src/github.ts` to avoid duplication.

### 6.3 Prompt templates

Path: `prompts/` in the Shopfloor repository.

Plain markdown files with `{{placeholder}}` variables substituted at runtime by a small helper (or inline in each stage job using `envsubst`-style substitution). Shipping prompts as files (rather than hardcoded strings in action code) means they can be iterated on independently and diffed over time.

| File | Used by |
|---|---|
| `prompts/triage.md` | Triage stage |
| `prompts/spec.md` | Spec stage |
| `prompts/plan.md` | Plan stage |
| `prompts/implement.md` | Implementation stage |
| `prompts/review-compliance.md` | Review stage, `compliance` matrix cell |
| `prompts/review-bugs.md` | Review stage, `bugs` matrix cell |
| `prompts/review-security.md` | Review stage, `security` matrix cell |
| `prompts/review-smells.md` | Review stage, `smells` matrix cell |

Each prompt begins with a rigid **system prompt block** that establishes:

1. The stage's single purpose.
2. What the agent may produce (file contents, commits).
3. What the agent may NOT do (post comments, open PRs, modify labels, call any non-allowlisted tools).
4. The structured output contract the agent must return.

This is belt-and-suspenders with the tool allowlist, but costs nothing and makes the contract explicit for anyone maintaining the prompts.

### 6.4 The Shopfloor MCP server

Path: `mcp-servers/shopfloor-mcp/index.ts`.

A small (50-100 line) MCP server implementation using `@modelcontextprotocol/sdk`. Namespaced as `shopfloor`. For v0.1, exposes exactly one tool:

**Tool: `update_progress`**

- **Full MCP name:** `mcp__shopfloor__update_progress`
- **Input:** `{ body: string }`
- **Behavior:** Calls `PATCH /repos/:owner/:repo/issues/comments/:comment_id` with the supplied body, replacing the contents of the tracking comment identified by `SHOPFLOOR_COMMENT_ID`.
- **Idempotent:** The tool replaces the comment body wholesale on every call. No appending, no history. The agent owns the structure of the comment (typically a markdown checklist).
- **Fails loudly if `SHOPFLOOR_COMMENT_ID` is not set**, so misconfigurations surface immediately in the workflow logs.

**Extensibility:** Future tools (`notify_slack`, `record_metric`, `request_review`, etc.) live in the same server, exposed as additional `mcp__shopfloor__<tool>` entries. No rename or rewiring needed. The `shopfloor` namespace is the stable extension point.

The server is only loaded (via `--mcp-config`) for the implementation stage. Triage, spec, and plan stages do not load it because they do not use it.

### 6.5 Relationship to `claude-code-action`

Shopfloor is a pure consumer. It never forks `claude-code-action`, never vendors its source, never patches it. Every customization Shopfloor makes flows through:

- `prompt:` input (agent mode, no tag-mode tracking comment)
- `claude_args` (model, max turns, allowed tools, system prompt, json-schema, MCP config)
- `settings:` (when complex settings like hooks are needed)
- `additional_permissions` (for CI log access in the impl stage, when needed)

This invariant is load-bearing. It means Shopfloor v0.1 gets every `claude-code-action` bug fix and feature for free on version bumps, and it means the `claude-code-action` team has zero interface obligation to us.

If we ever *need* a feature `claude-code-action` doesn't have (for example, the `CLAUDE_COMMENT_ID` env var in agent mode), the right path is an upstream PR, not a fork.

## 7. Data flow walkthroughs

### 7.1 Large-complexity ticket, happy path

1. Human opens issue #42 titled "Add GitHub OAuth login" with a detailed body.
2. Shopfloor workflow fires on `issues.opened`. Router reads: no `shopfloor:*` labels, issue is open, not a PR. Decides `stage=triage`. Outputs the stage and issue context.
3. Triage job runs. Bootstrap step creates any missing Shopfloor labels. Triage agent runs with the triage prompt and reads the issue body via its `Read` and `Grep` tools (the issue body is included in the prompt directly; the agent may also read repo files for context). Returns `{status: "classified", complexity: "large", rationale: "Affects auth flow, DB migration, and UI; nontrivial scope"}`.
4. Router post-steps post a comment on #42: "Classified as `large`. Drafting spec next." Apply `shopfloor:large` and `shopfloor:needs-spec`. Remove `shopfloor:triaging`.
5. The label change fires the workflow again. Router reads `needs-spec`, outputs `stage=spec`. Spec job runs. Agent creates branch `shopfloor/spec/42-github-oauth-login`, writes `docs/shopfloor/specs/42-github-oauth-login.md`, commits, pushes. Returns structured output.
6. Router post-step opens PR #43 from `shopfloor/spec/42-github-oauth-login` to `main`. PR body gets the agent-authored content plus the metadata block. Applies `shopfloor:spec-in-review` to issue #42, removes `shopfloor:needs-spec`. Posts "Spec drafted: #43" on #42.
7. Human reviewer reviews PR #43, leaves line comments, submits a formal `Request changes` review.
8. Shopfloor workflow fires on `pull_request_review.submitted`. Router reads: PR has `Shopfloor-Stage: spec` metadata, the review state is `changes_requested`. Outputs `stage=spec, revision_mode=true, review_comments_json=<...>`.
9. Spec job runs again. Agent reads the existing spec file, reads the review comments from the prompt input, pushes new commits addressing them. Returns structured output with a short summary for the PR comment.
10. Router post-step posts "Revisions pushed. Please re-review." on PR #43. No label changes.
11. Human approves PR #43, merges it to `main`. The spec file now lives at `main:docs/shopfloor/specs/42-github-oauth-login.md`.
12. Workflow fires on `pull_request.closed, merged=true`. Router reads: PR has `Shopfloor-Stage: spec`, merged is true. Outputs `stage=none` for this event but triggers `handle-merge` logic: flips issue #42 from `shopfloor:spec-in-review` to `shopfloor:needs-plan`, posts "Spec merged. Moving to planning stage." on #42.
13. Label change fires again. Router outputs `stage=plan`. Plan job runs on branch `shopfloor/plan/42-github-oauth-login`, same pattern as spec, produces `docs/shopfloor/plans/42-github-oauth-login.md`.
14. Plan PR opened as #44, reviewed, merged. Issue #42 transitions to `shopfloor:needs-impl`.
15. Implementation stage: router pre-work creates branch `shopfloor/impl/42-github-oauth-login`, opens placeholder PR #45 with body "Shopfloor is drafting this PR now...", creates the initial progress comment on #45, writes the MCP config file, starts the impl agent.
16. Impl agent reads the plan file, writes out an initial TODO checklist via `update_progress`, works through the tasks, updates the checklist as each task completes. Commits code changes, pushes. Returns structured output.
17. Router post-step updates PR #45's body with the agent's final narrative and finalizes the progress comment to "Implementation complete, handing off to agent review." It then checks for the `shopfloor:skip-review` label on the PR or issue; absent here, so it applies `shopfloor:needs-review` to issue #42. (If the skip-review label had been present, it would instead apply `shopfloor:impl-in-review` and exit, bypassing the review stage entirely.)
18. The push in step 16 also fires `pull_request.synchronize` on PR #45. The router reads the event, sees `Shopfloor-Stage: implement`, checks the skip conditions (none apply), and dispatches the review stage. Iteration counter currently 0 in the PR body metadata (or absent, treated as 0).
19. Review stage runs. The matrix dispatches four jobs: `review-compliance`, `review-bugs`, `review-security`, `review-smells`, each running `claude-code-action` in agent mode with its stage-specific prompt and a read-only tool allowlist. They run in parallel on separate runners.
20. After all four matrix cells finish, the aggregator job runs. It reads the structured outputs, dedupes, filters by confidence >= 80, and:
    - **Clean path (first-iteration happy):** The filtered list is empty. Aggregator posts an approving review ("Shopfloor agent review: clean across all four reviewers"), sets commit status `shopfloor/review` to `success`, applies `shopfloor:review-approved` to issue #42, and removes `shopfloor:needs-review`. Iteration counter frozen at 1.
    - **Issues path:** The filtered list has one or more comments. Aggregator posts a Request Changes review with all comments batched, sets commit status `shopfloor/review` to `failure`, applies `shopfloor:review-requested-changes`, increments iteration counter to 1 in the PR body metadata. Continues at step 22.
21. (Clean path continuation) Human reviews PR #45, approves, merges. Workflow fires on merge. Router flips #42 to `shopfloor:done`, closes #42 with a reference to #45, posts "Implementation merged. Thanks." on the (now closed) issue. Pipeline complete.
22. (Issues path) The aggregator's Request Changes review fires `pull_request_review.submitted` with `changes_requested`. Router reads the event, sees the review author matches the Shopfloor bot identity, dispatches the impl stage in revision mode with the review comments as input. Impl agent pushes revision commits. `pull_request.synchronize` fires, review stage runs again, iteration counter increments, and so on.
23. The loop terminates when either:
    - A review iteration is clean (goes to step 21), or
    - The iteration counter exceeds `max_review_iterations`. In that case the aggregator skips posting the next Request Changes review, applies `shopfloor:review-stuck` to #42, posts "Iteration cap reached after N rounds. Please take over this PR." on PR #45, and sets the commit status to `failure` with a "cap reached" description. A human then either merges anyway, removes the label to force another round, or takes over the PR manually.

### 7.2 Medium-complexity ticket

Same as 7.1 except triage returns `complexity: "medium"`. The router skips the spec stage entirely: it applies `shopfloor:medium` and `shopfloor:needs-plan` directly. The plan stage reads the issue body (and any clarifications from triage) as its primary input, with no spec file to read. Everything from step 13 onward is identical.

### 7.3 Quick-complexity ticket

Same as 7.1 except triage returns `complexity: "quick"`. Router applies `shopfloor:quick` and `shopfloor:needs-impl` directly, skipping both spec and plan. The impl stage reads the issue body directly as its spec-and-plan combined. Everything from step 15 onward is identical. One PR total for quick tickets.

### 7.4 Triage requests clarification

1. Issue #50 opened with an underspecified body.
2. Triage agent runs, returns `{status: "needs_clarification", clarifying_questions: ["Which OAuth provider?", "Should existing accounts be auto-linked?"]}`.
3. Router posts a single comment on #50 containing the questions in a formatted markdown block. Applies `shopfloor:awaiting-info`, removes `shopfloor:triaging`. Pipeline halts.
4. Human replies in a comment with answers.
5. A maintainer removes `shopfloor:awaiting-info`. The router reads this as a label-removed event, decides `stage=triage`, re-runs triage. The triage agent sees the issue body plus the new comments, classifies, and proceeds as in 7.1.

### 7.5 Aborted pipeline

1. Issue #42 is at `shopfloor:plan-in-review`, PR #44 open.
2. Human closes issue #42, deciding the feature isn't worth it.
3. Workflow fires on `issues.closed`. Router reads: issue is closed. Outputs `stage=none`. No action taken.
4. PR #44 remains open but orphaned. A human closes it manually if desired. Shopfloor does not clean it up (we err on the side of not destroying work).

### 7.6 Stage failure

1. Plan agent crashes halfway through with a timeout.
2. The plan job's final `if: failure()` step (from `router/report-failure/`) posts a diagnostic comment on PR #44: "Plan stage failed. See [workflow run](link). You can retry by removing the `shopfloor:failed:plan` label or by adding `shopfloor:revise`." Applies `shopfloor:failed:plan` to the issue (or PR).
3. Human reads the workflow logs, decides the failure was transient, removes the `shopfloor:failed:plan` label.
4. Router reads this as a label-removed event, decides `stage=plan`, re-runs. No auto-retries ever; humans decide.

## 8. User-facing install experience

### 8.1 The three install steps

**Step 1: Set an auth secret** (30 seconds). User adds one of: `ANTHROPIC_API_KEY`, `CLAUDE_CODE_OAUTH_TOKEN`, or one of the cloud provider secrets. If the user already runs `claude-code-action`, this step is already done.

**Step 2: Install the Claude GitHub App** (optional, 60 seconds). One click at `github.com/apps/claude`. Optional but recommended so bot comments appear as `claude[bot]` instead of `github-actions[bot]`.

**Step 3: Add the caller workflow** (60 seconds). Paste the following into `.github/workflows/shopfloor.yml`:

```yaml
name: Shopfloor
on:
  issues:
    types: [opened, edited, closed, labeled, unlabeled]
  issue_comment:
    types: [created]
  pull_request:
    types: [opened, synchronize, closed, labeled, unlabeled]
  pull_request_review:
    types: [submitted]
  pull_request_review_comment:
    types: [created]

jobs:
  shopfloor:
    uses: niranjan94/shopfloor/.github/workflows/shopfloor.yml@v1
    permissions:
      contents: write
      pull-requests: write
      issues: write
      id-token: write
      actions: read
      statuses: write
      checks: read
    secrets: inherit
```

Commit, push, done.

### 8.2 First-run bootstrap

On the first Shopfloor workflow run in a repository, the router performs an idempotent bootstrap:

1. Checks which `shopfloor:*` labels exist on the repository.
2. Creates any missing labels with sensible colors:
   - `shopfloor:triaging` (yellow)
   - `shopfloor:awaiting-info` (orange)
   - `shopfloor:quick`, `shopfloor:medium`, `shopfloor:large` (blue)
   - `shopfloor:needs-spec`, `shopfloor:needs-plan`, `shopfloor:needs-impl` (purple)
   - `shopfloor:spec-in-review`, `shopfloor:plan-in-review`, `shopfloor:impl-in-review` (cyan)
   - `shopfloor:needs-review`, `shopfloor:review-requested-changes` (cyan)
   - `shopfloor:review-approved` (green)
   - `shopfloor:review-stuck` (red)
   - `shopfloor:skip-review` (gray)
   - `shopfloor:done` (green)
   - `shopfloor:failed:triage|spec|plan|implement|review` (red)
   - `shopfloor:revise` (gray)
3. Posts a first-contact comment on the triggering issue introducing Shopfloor.

This runs on every workflow invocation but is idempotent: existing labels are left alone. The cost is one GitHub API call on the happy path.

No configuration file is created. No branch is created. No files are committed. The user's repository looks identical to before until the first spec PR is drafted.

### 8.3 Troubleshooting (to appear in README)

- **Branch protection requiring signed commits:** Enable `ssh_signing_key` via Shopfloor's secret passthrough.
- **Branch protection requiring linear history:** Shopfloor uses merge commits by default; switch to rebase/squash merges in the caller workflow's PR merge policy, or adjust your branch protection.
- **CODEOWNERS review requirements:** Add the Claude GitHub App (or custom Shopfloor App) as a codeowner for paths Shopfloor will modify.
- **Private GitHub Enterprise:** Set `ANTHROPIC_BASE_URL` and related env vars at the job level in the caller workflow.
- **Custom PR template conflicting with metadata block:** The metadata block is always appended to the bottom of the PR body; user templates are preserved above it.

| Permission | Level | Why |
|---|---|---|
| `contents` | `write` | Create branches, push commits, read repository files. |
| `pull-requests` | `write` | Open PRs, update PR bodies, read review comments, add labels. |
| `issues` | `write` | Read issues and comments, post comments, manage labels, close issues. |
| `id-token` | `write` | Used by `claude-code-action` for OIDC exchange with Bedrock/Vertex/Foundry. Harmless on Anthropic API setups. |
| `actions` | `read` | Read workflow run metadata for diagnostic links and optional CI log access. |
| `statuses` | `write` | Required by the review stage aggregator to create and update the `shopfloor/review` commit status on impl PRs. |
| `checks` | `read` | Read existing status checks (e.g. to avoid re-running the review stage if a prior Shopfloor review already passed for the current HEAD SHA). |

Explicitly **not** required:

| Permission | Why not |
|---|---|
| `checks: write` | Shopfloor uses commit statuses, not check runs. `checks: read` is sufficient. |
| `workflows: write` | Deliberately avoided. The impl agent must not be able to rewrite `.github/workflows/*`. Users who want this can add it themselves via `additional_permissions`. |
| `discussions`, `packages`, `pages`, `security-events` | Unused. |

### 9.2 GitHub App identity (optional)

Default: `claude-code-action` uses the Claude GitHub App (if installed) or `GITHUB_TOKEN` (if not). Bot comments appear as `claude[bot]` or `github-actions[bot]`.

Custom: users can register their own GitHub App and pass `github_app_id` + `github_app_private_key` as secrets. Required app permissions: `Contents: read/write`, `Issues: read/write`, `Pull requests: read/write`, `Metadata: read`. Subscribed events: `Issues`, `Issue comments`, `Pull requests`, `Pull request reviews`, `Pull request review comments`.

### 9.3 Single-bot visual experience

To avoid showing two bot identities (`claude[bot]` for agent-owned work, `github-actions[bot]` for router-owned work), the router reuses the token `claude-code-action` mints (exposed via the `github_token` output of the upstream action) for its own GitHub API calls when available. Fall back to `GITHUB_TOKEN` only on diagnostic-failure paths where the upstream action may not have successfully started. Net effect: one bot identity in the timeline on the happy path.

**Threading the token across jobs.** GitHub Actions jobs do not share process state, so the minted token must be explicitly propagated. The stage job that invokes `claude-code-action` (e.g., the `implement` job) captures the upstream action's `github_token` output and re-exports it as a job output (e.g., `impl_github_token`). Downstream jobs that need to act as the same bot identity (e.g., the review matrix jobs, the aggregator job, router post-steps) read the token from the upstream job's outputs via `needs.implement.outputs.impl_github_token`. Jobs that run before any `claude-code-action` invocation in the workflow (e.g., the `route` job, the `triage` job before its claude-code-action step) fall back to `GITHUB_TOKEN`. This is a small amount of explicit plumbing in the reusable workflow but keeps the router's GitHub API calls under the same bot identity as the agents'.

## 10. Agent and router boundary

**The bright line:** anything that changes the state machine is performed by the router; anything that produces user-visible content is performed by the agent via structured output, which the router then templates into comments, PR bodies, and files.

### Agent responsibilities

- Read issue body, comments, PR body, review comments, repository files, diffs (review stage only).
- Write spec, plan, and implementation file contents (non-review stages only; review agents are read-only).
- Create and push commits on the working branch (non-review stages only).
- Update the progress tracking comment via `mcp__shopfloor__update_progress` (impl stage only).
- Return structured output per the stage's JSON schema.
- For the review stage: the structured output includes a `comments` array with line-anchored inline review comments. The agent does **not** post these directly; the aggregator (a router-owned helper) reads the structured output and posts a single GitHub review with all surviving comments batched via the Reviews API.

### Router responsibilities

- Read the event payload, infer the stage, decide whether to run.
- Create missing Shopfloor labels on first run.
- Open stage PRs with agent-supplied title/body and metadata block.
- Apply and remove `shopfloor:*` labels.
- Post status-transition comments ("Classified as large", "Spec drafted: #43", etc.) from templates.
- Post failure diagnostic comments (in `if: failure()` steps).
- Create and finalize the initial progress tracking comment (impl stage only).
- Write the MCP config file for the impl stage.
- Close the origin issue when the impl PR merges.
- Aggregate the review stage's matrix outputs, dedupe and filter comments, and post the single batched GitHub review (APPROVE or REQUEST_CHANGES) via the Reviews API.
- Manage the `shopfloor/review` commit status on impl PRs (`pending` / `success` / `failure` / `error`).
- Track and increment the impl↔review iteration counter in the impl PR body metadata.
- Apply `shopfloor:review-stuck` and post a takeover comment when `max_review_iterations` is exceeded.

### Explicit prohibitions on the agent

- **No comment posting.** No `mcp__github_comment__*`, no `mcp__github__add_issue_comment`, no `mcp__github_inline_comment__*`. Triage clarifying questions are returned as structured output; the router posts them.
- **No PR creation or modification.** The router opens every PR and writes every PR body.
- **No label manipulation.** State transitions are the router's exclusive domain.
- **No issue creation, closing, or editing** (other than via the structured output surface).

Enforcement:

1. **Primary:** the `--allowedTools` allowlist omits every comment-posting and PR-mutating MCP tool. The LLM sees them as unavailable.
2. **Secondary:** the system prompt block in every stage's `prompts/*.md` spells out these prohibitions explicitly.
3. **Tertiary:** claude-code-action is invoked in agent mode (by setting `prompt:`), which disables the auto-tracking-comment behavior entirely.

## 11. Configuration surface

The caller workflow's `with:` block is the entire user-facing config surface. No config files, no env files, no CLI. Every knob is a typed input on the reusable workflow.

When more complex configuration is needed (hooks, per-stage custom system prompts, overrides to `settings:`), users pass them through as `claude_args` or a serialized `settings:` JSON string. Shopfloor forwards these verbatim to `claude-code-action`.

If a user's configuration needs exceed what fits comfortably in the caller workflow's `with:` block, that is a signal that Shopfloor should reconsider its defaults, not a signal to add a config file format. YAGNI.

## 12. MCP injection and coexistence

Shopfloor's `shopfloor` MCP server is injected via a single additional `--mcp-config` flag for the impl stage only. It coexists with:

- `claude-code-action`'s built-in `github_comment`, `github_file_ops`, `github_inline_comment`, `github_actions_server`, and `github` MCP servers.
- The user's repository-level `.mcp.json` (auto-loaded by the Claude CLI due to `claude-code-action`'s forced `enableAllProjectMcpServers: true`).
- Any additional `--mcp-config` flags the user passes via `claude_args`.

Coexistence rules:

- **Name uniqueness.** Shopfloor's server is named `shopfloor`, which no sensible user will have in their `.mcp.json`.
- **No file replacement.** Shopfloor writes its config to `$RUNNER_TEMP/shopfloor-mcp.json`, outside the user's checkout. `claude-code-action`'s restore-from-base-branch security mechanism does not touch `$RUNNER_TEMP` contents.
- **Allowlist gating.** User-defined MCP servers are loaded but their tools are only callable if they appear in the stage's `--allowedTools`. Shopfloor's per-stage allowlists exclude user MCP tools by default. Users who want to grant their own MCP tools to the impl agent use the `additional_tools` input.

## 13. Error handling

### Agent failure

Each stage job has a final step `if: failure() || cancelled()` that runs `router/report-failure/`. This step:

1. Posts a diagnostic comment on the appropriate artifact (origin issue for triage failures, stage PR for spec/plan/impl failures), including a link to the workflow run.
2. Applies `shopfloor:failed:<stage>` to the issue.
3. Returns gracefully. No re-raise.

No automatic retries. Agents are not flaky network calls; a failure usually means something needs human attention. Humans remove the failed label to retry.

### Router failure

The router itself is simple enough that failures indicate genuine bugs. It exits with a non-zero code and the job fails. Since there is no stage label change in this case, the workflow does not advance, and re-running the workflow (via `rerun-failed-jobs`) is a safe retry.

### Review stage failures

- **A single matrix cell fails:** The aggregator job has `if: always()` and still runs. It treats the failed cell as "no findings from that reviewer" and proceeds with the remaining outputs. The failure is logged so humans can see which reviewer broke. The commit status still gets set correctly.
- **All matrix cells fail:** Aggregator runs with no inputs, treats the review as "inconclusive," sets commit status to `error` with a description pointing to the workflow run, applies `shopfloor:failed:review` to the issue. Does not post a Request Changes review (which would retrigger impl for no reason).
- **Aggregator itself fails:** The final `if: failure()` step posts a diagnostic comment on the impl PR, applies `shopfloor:failed:review`, sets the commit status to `error`. Humans intervene.
- **Iteration cap exceeded:** Not a failure; the aggregator deliberately stops the loop. See section 5.5.3 step 10.

### Missing or malformed state

If the router detects inconsistent state (e.g., two state labels applied simultaneously, a `needs-plan` label with no merged spec file, a branch-naming mismatch with the PR body metadata), it emits `stage=none` and posts a diagnostic comment on the issue asking a human to fix the state. Shopfloor never attempts to auto-repair state.

### Concurrency conflicts

`concurrency: shopfloor-issue-${{ issue_number }}` with `cancel-in-progress: false` serializes all events for a given issue. If an event fires while another is running, it queues. This prevents the agent from racing against itself on the same branch.

### Rate limits and timeouts

- Each stage job has a per-stage `timeout-minutes` default (triage: 10, spec: 20, plan: 30, impl: 60), user-overridable via inputs.
- GitHub API rate limits are handled by octokit's built-in retry; the router does not implement its own backoff.
- Anthropic API rate limits bubble up as `claude-code-action` failures, handled by the normal failure path.

## 14. Security considerations

### Attacker-controlled inputs

- **Issue bodies, comments, PR descriptions, review comments.** These are all attacker-controlled on public repositories. Every agent prompt must be built with the assumption that these contents may contain prompt injections. Mitigations:
  - System prompt instructions in `prompts/*.md` explicitly tell the agent to treat user content as data, not instructions.
  - Tool allowlists are minimal per stage.
  - The agent cannot post comments, open PRs, or change labels even if prompt-injected to try.
  - Triage is the only stage that interacts with an issue by reading its comments; downstream stages read the spec and plan files (which are human-approved by the time the downstream stage runs).
- **PR-authored `.mcp.json`, `.claude/`, `CLAUDE.md`.** On PR branches, these are restored from the base branch by `claude-code-action`'s `restore-config.ts` before the agent runs. Shopfloor inherits this protection.
- **Merged spec and plan files as trusted inputs.** Once a spec PR or plan PR is merged, its markdown file is treated as a trusted input by downstream stages (plan reads spec, impl reads plan). `restore-config.ts` does NOT cover `docs/shopfloor/specs/` or `docs/shopfloor/plans/` because they are not in the protected path list. This means a carefully crafted issue could lead the spec agent to write prompt-injection payloads into the spec file, which would then be read by plan and impl as if authoritative. **The only mitigation is human review at the merge boundary.** Humans reviewing spec and plan PRs must read the diffs as if they were code, not glance at them as documentation. This is a known residual risk for v0.1; closing it further would require either a second sanitising agent pass before merge or restricting spec/plan content to a constrained schema.

### Secrets

- Shopfloor does not read any secrets from the environment directly. All secrets flow through `secrets:` inputs to the reusable workflow and are forwarded to `claude-code-action`.
- The Shopfloor MCP server receives `GITHUB_TOKEN` in its env block; that token is scoped to the workflow's permissions.

### The `workflows: write` prohibition

The impl agent does not have `workflows: write` permission, and the default `impl_bash_allowlist` does not include shell commands that could modify `.github/workflows/*`. An agent cannot rewrite its own orchestration, cannot add new workflows, and cannot disable Shopfloor's routing.

### Supply chain

Shopfloor pins `claude-code-action` to a specific tag in its reusable workflow. Version bumps are explicit in the Shopfloor repo and user-visible in the caller workflow (since users reference `@v1` and we document what `v1` includes).

## 15. Testing strategy

### 15.1 Router unit tests (highest priority)

The router's state logic in `router/src/state.ts` is pure: inputs in, decision out. We will have comprehensive unit tests using `vitest` (or `bun:test`) that cover:

- Every valid event type (`issues.opened`, `issues.labeled`, `issues.unlabeled`, `issue_comment.created`, `pull_request.opened`, `pull_request.closed` with merged=true, `pull_request_review.submitted`, etc.).
- Every valid label state combination.
- Every valid transition from section 4.
- Invalid combinations (two state labels, malformed branch names, PR body metadata mismatches, closed issues mid-pipeline).
- Abort paths.
- Revision triggers.

Target: 100% line coverage for `state.ts`. This is the one piece of code we cannot afford to break.

### 15.2 GitHub adapter tests

`router/src/github.ts` is tested with mocked octokit (`@octokit/rest` fixtures). Verifies API call shapes: label names, comment bodies, branch creation calls, PR creation calls.

### 15.3 Prompt snapshot tests

`prompts/*.md` rendered against fixture contexts produces stable outputs. Snapshot tests catch accidental template breakage.

### 15.4 MCP server tests

`mcp-servers/shopfloor-mcp/index.ts` tested by spawning it as a subprocess, sending MCP `tools/call` messages, verifying PATCH calls to a mock GitHub API.

### 15.5 Review aggregator unit tests

The `router/aggregate-review/` helper is pure logic that takes a bag of structured reviewer outputs and produces a single GitHub review payload plus a commit status plus a label transition. It is tested extensively with:

- Fixture bags simulating all four reviewers returning `clean`.
- Fixture bags simulating issues from all four reviewers with varying confidence scores, verifying the filter and dedupe logic.
- Duplicate comments on the same `{path, line}` from multiple reviewers, verifying the dedupe heuristic picks the higher-confidence version.
- Below-threshold comments verifying they are dropped.
- Matrix-cell-failure fixtures (1, 2, 3, or all 4 cells failed) verifying graceful degradation.
- Iteration cap scenarios: exactly-at-cap, over-cap, under-cap.
- Skip-condition scenarios: draft PR, closed PR, no changed files, already-reviewed-at-SHA.

Target: 100% branch coverage on the aggregator. It is the one piece of review-stage logic we cannot afford to break.

### 15.6 End-to-end integration test

A canned test harness that:

1. Spins up a local GitHub-Mock server (e.g., `mockoon` or a bespoke Express app) that simulates the GitHub REST API endpoints Shopfloor depends on.
2. Replaces `claude-code-action` with a stub that reads a fixture file and returns canned structured output.
3. Drives the state machine by calling the router and helper actions directly with fixture event payloads, rather than trying to run the full reusable workflow under `act`. `act` does not faithfully emulate matrix jobs, cross-job outputs, reusable-workflow semantics, the Reviews API, or branch protection, so the e2e harness operates at the action level, not the workflow level. A thin orchestrator in the harness plays the role of the reusable workflow's job wiring.
4. Asserts the sequence of API calls, label changes, file writes, and commit status transitions matches expectations.

One e2e test per happy path (large, medium, quick), plus one per error path (triage asks for clarification, stage failure, abort). Review-stage e2e coverage: one test where the first review passes on iteration 1; one test where review requests changes on iteration 1 and passes on iteration 2; one test where review fails all 3 iterations and the loop caps out with `shopfloor:review-stuck`.

### 15.7 Dogfood

Shopfloor builds Shopfloor. Before v0.1 ships, we use Shopfloor on the Shopfloor repository itself for at least one non-trivial feature. If we cannot use it to build it, it is not ready.

## 18. Open questions and risks

None of these are blockers for writing the implementation plan; they are items to revisit during implementation.

- **Progress comment location for quick tickets.** Quick tickets go straight to impl with no spec/plan PRs. The progress comment lives on the impl PR, which is opened immediately in the impl stage pre-work. No change from the large/medium flow.
- **What if the impl agent's `changed_files` output disagrees with what actually got committed?** Not a blocker: the PR diff is the source of truth. The `changed_files` output is informational only, used for summary comments.
- **Rate of agent-produced revisions.** If a human leaves many rapid-fire `Request changes` reviews, the agent could chain revisions indefinitely. Not a v0.1 problem (it is rate-limited by human review pace), but worth monitoring.
- **Branch naming slug derivation.** How we derive `<slug>` from the issue title (kebab-case, first N words) needs a small helper function and a decision on max length. Default: first 5 words of the title, kebab-case, max 40 characters. Revisit if it produces ugly names.
- **Upstream PR for `CLAUDE_COMMENT_ID` env var in agent mode.** Not needed for v0.1 (we use our own MCP server), but a good contribution to make later so future Shopfloor versions could optionally reuse claude-code-action's tracking comment plumbing if desirable.
- **Metadata block vs PR body editing.** PRs opened by Shopfloor always append `Shopfloor-Issue: #<n>` and `Shopfloor-Stage: <stage>` at the bottom, including `Shopfloor-Review-Iteration: <n>` on impl PRs. If a human edits the PR body and removes these, the router falls back to branch name parsing for the stage identifier; for the iteration counter, it resets to 0 (worst case: the loop can run one extra time).
- **Reviewer disagreement.** Two reviewers may flag overlapping issues with different interpretations (e.g., the compliance reviewer says "this is fine under the style guide" and the smells reviewer says "this is still ugly"). The dedupe heuristic merges them when their text is similar; otherwise both survive. This is intentional: the human merging the PR sees both perspectives rather than having one silently win.
- **Human reviewers coexisting with agent reviewers.** If a human reviews an impl PR while the review stage is running, the result is two reviews on the same PR. This is fine (GitHub supports multiple reviews). The router distinguishes human reviews from Shopfloor reviews by actor identity when deciding whether to retrigger impl.
- **Reviewer drift under long diffs.** Extremely large impl PRs may produce truncated diffs in the reviewer prompts and noisy output. Mitigation is the iteration cap, not a special case. If it becomes a real problem we add a "split large PRs" concept in a later version.

## 19. Summary

Shopfloor v0.1 is a reusable GitHub workflow that drives a five-stage, human-gated pipeline (triage → spec → plan → implement → agent review) by orchestrating `claude-code-action` across stages. State lives in GitHub (labels, branches, PR metadata, committed markdown files). A deterministic TypeScript router owns every state transition; claude-code-action agents in agent mode own every piece of content. A small Shopfloor-namespaced MCP server gives the implementation agent a single live progress comment to update. The review stage runs a 4-way matrix of specialised reviewer subagents (compliance, bugs, security, code smells) in parallel, and a router-owned aggregator merges their outputs into a single batched GitHub review plus a commit status. The impl↔review loop has a configurable iteration cap to guarantee termination. The install is one workflow file. Upgrades are one version bump.

The scope is deliberately narrow: no sandboxes, no telemetry, no dashboards, no external integrations, no CLI, no config files. Every feature beyond the core pipeline is explicitly deferred. If Shopfloor v0.1 ships and survives contact with real use, we revisit the deferred features with data instead of guesses.
